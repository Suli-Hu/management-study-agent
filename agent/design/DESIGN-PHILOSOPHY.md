# Design Philosophy · sususu

> 这份文档不告诉你颜色的 hex 值是多少，也不告诉你按钮该多高。  
> 这份文档告诉你：**当你不知道一个东西该怎么设计时，怎么决定。**

---

## §0 这份文档不是什么

| 你想找 | 应该看 |
|---|---|
| Token 数值（颜色 OKLCH、字号、间距、圆角） | `study.sususu.org/design-system` mirror page，或 `v2/src/styles/tokens.css` |
| 单个组件的尺寸 / 状态 / 接口 | `design/COMPONENTS.md`（待建） |
| 单个页面的 layout 规格 | `design/pages/*.md`（待建） |
| **判断规则**："这里该用红色还是灰色？" "这里该是 pill 还是矩形？" | **本文** |

**优先级**：当本文与具体组件文档冲突时，**本文赢**。组件文档是这些原则的"应用例"，不是凌驾于原则之上的特例。

## §1 三条铁律

> 这三条压过本文档其他所有内容。设计争议时，先翻这一节。

### 铁律 1 · 形状 > 颜色

语义优先靠**形状 / 结构 / 位置 / 字重**传达，颜色只做最后一层强化。

**为什么**：颜色在以下场景全部失效——色盲（约 8% 男性）、暗色模式色相偏移、运营随手配色撞色（"营销 chip 红、tag 战略红"）、打印、深色背景/浅色背景反转。形状不会失效。

**应用**：
- 一个元素是 pill 还是矩形 = **它有没有交互**（不变）
- 一个元素是大字还是小字 = **它的层级**（不变）
- 一个元素是红色还是灰色 = **它的状态**（可变 / 可叠加在前两者之上）

### 铁律 2 · Token 守则赢

`tokens.css` 文件头注释里写的 "用法守则" 是**契约**，不是建议。当组件层在违反它，**修组件，不要改 token**。

**为什么**：token 是底层公约。一旦某个组件可以"破例使用"，所有组件都会跟着破例（chip 1-18 历史已经验证过这条）。token 一旦松动，整个系统就回到"每个组件独立配色"的混乱前态。

**当前已知违反**（必须改）：
- tokens.css 写明 "tag 色仅用于学派归属（小圆点 / 角标 / sparkline）"——但 KP 详情页 chip 用 tag 色做实心填充。**chip 必须改**，不是 token 必须改。
- tailwind.config.ts 的 `accent.{ob, classic, strategy, warning}` 是 v1 遗产、与 tokens.css 的 OKLCH tag 系统并存——这是**双色源 bug**，工程需在一个 PR 里裁决（建议留 OKLCH 一套，废 tailwind accent；warning 接 `--s-danger`）。

### 铁律 3 · 不确定就选最少

每个新增的颜色 / 图标 / 阴影 / 装饰都要为自己**辩护**。默认是不加。

**判定句**：删掉它会丢失什么信息？  
- 答 "不会丢" → 删掉  
- 答 "会丢 X 信息" → 那 X 信息能不能用形状 / 排版 / 间距承担？能 → 删掉颜色。

**反例**：chip 行加红色实心 pill 想表达 "这是分类"——分类信息已经由"它在 chip 行的位置"和"它的文字内容（学派名）"承担了。颜色没传递新信息，只制造了焦点错位。删。

## §2 可交互 vs 不可交互

> chip 1-18 事故的核心：**用户分不清哪些能点、哪些不能点**。这一节是根治。

### 4 个交互信号

一个元素**可点**的视觉证据，按强度排序：

1. **cursor: pointer**（用户必须 hover 才知道，最弱）
2. **hover 状态变化**（背景变深 / 出现箭头 / 边框出现）
3. **形状语言**（pill 形 / 按钮形 / 链接下划线）
4. **明确装饰**（→ 箭头 / 外链图标 / "查看更多"文字）

### 规则

| 元素类型 | 必须具备的信号 | 严禁 |
|---|---|---|
| **可交互**（按钮 / 链接 / tab / chip-as-filter） | **至少 3 个** | 缺信号但又能点 |
| **不可交互**（label / chip-as-tag / 静态徽章） | **不超过 1 个**（cursor 必须为 default） | pill 形 + hover 变色但点了无反应 |

### 决策树："我做的这个元素该怎样？"

```
问 1：用户点它会发生什么？
  ├─ 什么也不发生 → 不可交互
  │    → 用：圆角矩形 (8px) / 圆点 / 纯文字
  │    → cursor: default，无 hover 状态变化
  │
  ├─ 跳转 / 触发 → 可交互
  │    → 用：pill (999px) / 按钮形 / 链接（下划线 or 箭头）
  │    → cursor: pointer，必须有 hover 反馈
  │
  └─ 不确定 → 让产品决定，不要凭感觉做
```

### 形状语义锁死

| 形状 | **锁定**含义 | 例子 |
|---|---|---|
| **pill (999px)** | 真交互（tab / toggle / 主按钮 / lang 切换） | active tab、`日本語/中文` 切换 |
| **圆角矩形 (8px)** | 静态标签 / 信息卡 / 不可点徽章 | 学派 chip、tag chip、tier 徽章 |
| **圆点 (4–6px)** | 分类标记（极小、装饰、不占语义带宽） | 列表项左侧学派色点 |
| **outline-only pill** | 弱可交互 / 当前态切换 | 已选中的语言切换 |
| **下划线文字 / 箭头** | 跳转链接 | "查看更多"、面包屑、KP 卡 hover |

**违反这张表 = 设计回归**。如果你想做的元素配不上任何一行，先问"它真的需要存在吗"。

## §3 颜色传达什么

颜色有 **4 种用途**，且**同一个元素只能承担一种**。这是 chip 灾难的根本原因——它同时想表达"分类"和"焦点"，而颜色只能做一件。

### 4 种用途 · 4 个色源

| 用途 | 含义 | 色源 | 例子 |
|---|---|---|---|
| **语义 (Semantic)** | 状态 / 进度 / 强度 | `--s-*`（success/warning/danger/info）/ `--i-*` 强度 / `--p-*` 进度 | toast 报错红、热力图深蓝、进度条墨黑 |
| **分类 (Category)** | "这属于哪一类"（学派、学科） | `--tag-*`（8 色板，永不动） | 列表里学派圆点、sparkline 配色 |
| **层级 (Hierarchy)** | "这比那个更重要" | `--text` / `--text-2` / `--text-3` 三级文字 + `--bg` / `--bg-soft` 两级底 | 主标题黑、次文字灰、placeholder 浅灰 |
| **装饰 (Decoration)** | 纯品牌 / 视觉节奏 | 受限：仅 `--primary`（墨黑）+ tag 色仅做点缀 | logo、focus 环 |

### "一个元素只能承担一种" 怎么应用

**chip 行 = 分类 (Category)** —— 学派属于"营销学科" / tag 属于"战略选择子类"，这是分类信息。

错误做法：用红色实心 pill 表达"这个 chip 重要"——这是**把分类元素叠加了焦点 (decoration / hierarchy)**，违反一元一用。

正确做法：
- chip 整体是**中性**的（灰底或白底 + 灰色边）——分类信息靠**位置**（chip 行的语义带）和**文字**（学派名）承担
- 颜色仅以**圆点 (4-6px)** 形式出现在 chip 文字左侧，承担"它属于哪个学派色"的次级信号
- 这样：chip 的分类 = 文字 + 小圆点，焦点 = 不参与（chip 不是焦点元素，焦点在主标题/正文）

### 冲突仲裁

**问题**：营销学派色 #EF4444 与 tag「战略选择」#EF4444 撞红。

**仲裁规则**（按优先级）：

1. **同元素同用途禁止撞色**——学派和 tag 都是"分类"，色板必须从同一池子分配，不能两个变量各自定义、互不通气
2. **跨用途撞色容忍**——但要求形状必须区隔。例：`--s-danger`（红、状态）和 `--tag-pink`（红、分类）色相接近是被允许的，因为前者只出现在 toast/icon、后者只出现在圆点，**形状把它们彻底分开**
3. **撞色发生时谁让位**：tag 色（L3）让位于 semantic 色（L2）。理由：semantic 色是**强信号**（用户必须立刻识别"出错了"），tag 色是**弱信号**（"哦这是哪一派"，可以用文字补救）

### 颜色禁区

- ❌ 用颜色制造焦点（应该用位置 / 大小 / 字重 / 留白）
- ❌ 用颜色区分两个**功能相同**的元素（用文字或图标）
- ❌ 给"装饰"用途超过 2 种颜色（品牌色滥用是 ai-slop 的标志）
- ❌ tag 色出现在大面积填充上（tag 是"指示色"不是"主体色"）

## §4 层级与焦点

### 焦点信号优先级

一屏只能有 1 个**主焦点**（用户视线第一个落点）。焦点的传达手段，按强度排序：

1. **位置** —— 在视觉重心 / 黄金分割点 / 唯一一个不对齐的元素
2. **大小** —— 显著大于周围元素（≥ 1.5x）
3. **字重** —— 加粗对比 regular
4. **留白** —— 周围空间显著大于其他元素
5. **颜色** —— 最弱，且仅在前 4 项不足以达成时使用

### 决策："我想强调这个元素，怎么做？"

```
不要先问 "用什么颜色"，先问 "它的位置/大小够突出吗"
  ├─ 位置已经突出 → 不需要任何额外手段
  ├─ 位置一般，但可以放大 → 加大字号或留白
  ├─ 已经够大但还想强调 → 加粗
  └─ 实在没办法（已经在边角又不能加大）→ 才考虑颜色，且优先 --primary（墨黑），不是 accent
```

**反例库**：在 sidebar 列表里给"今日"项加红色文字想强调它——错。"今日"已经在列表第一行（位置最强）+ 时间标签自带突出（字重）。再叠红色 = 焦点过载，反而冲淡其他重要信息（如 KP 标题）。

### 多焦点禁区

如果一屏出现 2 个以上"想抢焦点"的元素，**说明你需要重新思考信息架构**——不是该用什么颜色对比，是这个屏幕信息密度过高、应该拆分。

## §5 反例图鉴

> chip 1-18 的演化、KP 详情页历次返工、design-system 页历次重写——浓缩成 5 条教训。  
> 不是历史档案，是"以后再这样就拒绝合并"的负面规则。

### 反例 1 · "用 tag 色实心填充 chip"（chip 1-18 的根源）

**做错的事**：chip 用学派色 / tag 色作为实心背景填充，制造"看起来很重要"的视觉。  
**为什么错**：违反铁律 1（颜色越权传达"焦点"，但 chip 是分类元素）+ §3 一元一用 + §2 形状语义（实心彩色 pill 长得像可点的按钮，但点了无反应）。  
**正确做法**：chip 灰底 / 白底 + 8px 圆角矩形，学派色仅作 4-6px 圆点出现在文字左侧。

### 反例 2 · "用红色试图强调'今日学习'"

**做错的事**：sidebar 列表里给当日项加 `--s-danger` 文字色。  
**为什么错**：违反 §4 焦点信号优先级（位置已经够强，叠颜色 = 过载）+ 滥用 semantic 色（红色应该意味"危险/错误"，不是"今天")。  
**正确做法**：用字重 + 左侧 2px 墨黑 indicator bar，颜色保持 `--text`。

### 反例 3 · "design-system 页 hover 后变红"

**做错的事**："这是个跳转"用 hover 变红色文字提示。  
**为什么错**：违反 §3（红色 = semantic 错误状态，不该承担"链接 hover"）+ 形状语义（链接的 hover 应该是下划线 / 字色加深 / 出现箭头，不是变红）。  
**正确做法**：hover 时 `text-decoration: underline` 或文字色由 `--text-2` 变 `--text`，外加 → 箭头从 0 透明度淡入。

### 反例 4 · "实心彩色 chip + cursor:pointer + 点了无反应"

**做错的事**：chip 看起来像按钮（实心 + pill），鼠标变手型，但点击无反应（或仅是页面 reload）。  
**为什么错**：4 个交互信号占了 3 个（pill 形 + cursor + 视觉强对比），违反 §2"不可交互不超过 1 个信号"。用户花 3 秒确认"它真不能点"是设计的失败。  
**正确做法**：要么真的让它能点（变成 filter，点击后筛选当学派内容），要么把所有交互信号清零（圆角矩形 + cursor:default + 无 hover）。**两种都行，骑墙不行**。

### 反例 5 · "用颜色区分功能相同的两个元素"

**做错的事**：列表里 "新建" 按钮蓝色、"编辑" 按钮绿色、"删除" 按钮红色——三个都是 primary action。  
**为什么错**：颜色不传递信息（用户不会通过颜色识别功能，仍要读文字），却消耗了焦点预算。  
**正确做法**：所有 primary action 同款 `--primary` 墨黑按钮，仅"删除"等危险操作用 `--s-danger` 边框（不是填充），靠**文字**区分功能。

## §6 当你要加新东西时

> 在写代码 / 起新组件之前，先用这 10 个问题过一遍。任何一个答不上来，先停。

### 加新组件 / 新色 / 新形状的决策清单

1. **现有组件能解决吗？** 9 成的"新需求"是已有组件用错了或被产品当成了新功能
2. **它表达的是什么？** 语义 / 分类 / 层级 / 装饰 选一个，不能多选
3. **它能不能复用 token？** 不行 → token 缺什么、要不要补，单独提工单，**不在组件里写死**
4. **它可交互吗？** 是 → 至少 3 个交互信号；否 → 不超过 1 个
5. **它的形状属于 §2 形状词典哪一行？** 不在表里 → 是不是该用现有形状？
6. **它会跟现有元素撞色吗？** 用 §3 冲突仲裁规则查
7. **暗色模式下它怎么变？** 不能假设"色相反转就行"——`--tag-*` 在暗色下不变，`--text` 反转，semantic 色降饱和
8. **322px 极窄屏下它怎么换行？** chip 行不能横向溢出、按钮文字必须能截断
9. **a11y 怎么办？** 颜色对比 ≥ 4.5:1（正文）/ 3:1（大字）；触控目标 ≥ 44×44pt
10. **如果 5 年后回看这个决策，理由能不能用一句话说清楚？** 不能 → 还没想清楚，不要做

### 提交 PR 前的自查 3 问

1. 你新加的颜色 / 形状 / 装饰 —— 删掉它，会丢失什么信息？（铁律 3）
2. 它跟 tokens.css 注释的"用法守则"冲突吗？（铁律 2）
3. 它是 chip 1-18 反例库里的哪一种翻版吗？（§5）

## §7 修订原则

这份文档**会演化**，但有规矩：

- **新增条款** → 必须配反例（"为什么以前错"）+ 正例（"现在该怎样"），且不能与铁律冲突
- **修改铁律** → 需要 3 个具体场景证明铁律带来的成本 > 收益，且整份文档跟着重审
- **删除条款** → 标注 deprecated 但保留半年，让历史 PR 引用还能找到上下文

**不允许的修订**：
- "因为这次需求紧急，先破例" —— 设计文档不接受紧急破例。紧急的是该次需求，不是文档
- "因为大家都这么做" —— 别的产品做错的事不该是我们的依据
- "因为产品经理这么要"求 —— 产品需求驱动**功能**，不驱动**底层视觉规则**。规则一变，所有历史决策都要复议，成本远高于做一个折中方案

## 附录 A · 常见疑问

**Q：tokens.css 和 tailwind.config.ts 颜色系统不一致，我该信哪个？**  
A：tokens.css 是设计意图（OKLCH + 用法守则），tailwind accent 是 v1 遗产。提工单合并到 OKLCH 一套，过渡期内**新组件只用 tokens.css 变量**。

**Q：design-system 页里 chip 写的就是实心彩色，是不是已经合法？**  
A：不。design-system 页是"实现镜像"，不是"规范源"。规则源是本文 + tokens.css 注释。design-system 页要按本文修订。

**Q：日文/中文混排有特殊规则吗？**  
A：有。中文用 `PingFang SC`，日文优先 `Hiragino Sans`，font-stack 已在 tailwind.config 定义。但**字号、字重、行高保持一致**——不要给中文加大字号"补偿"，那是把中文当二等公民。

**Q：暗色模式下 tag 色不变会不会太亮？**  
A：tokens.css 注释明确"tag 色不变，保持识别一致性"——这是有意为之。tag 在暗色背景下是变得"更亮"，但因为它只用作小圆点 / sparkline（占面积小），不会刺眼。如果 tag 色被错用作大面积填充，那是**组件违反守则**，不是 token 该改。

**Q：什么时候可以打破规则？**  
A：当你能写出一份比本文更严密的反驳，且这份反驳被另一位设计师 review 通过。"我觉得不好看" / "客户喜欢" 不是理由。
